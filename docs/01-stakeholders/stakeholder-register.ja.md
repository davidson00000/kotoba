## 役割の選定理由（各エージェントがなぜそのスロットに適しているか）

- **S5 Claude Code → リードエンジニア**: エージェンティックなコーディング能力 + リポジトリ認識能力 = コードベース全体のアーキテクチャ文脈を保持するのに最適。
- **S6 Codex Desktop → Dev**: ローカル実行可、コーディング能力が高い、ブラウザ自動化の強みをここで消費する必要がない。
- **S7 Antigravity IDE → QA**: 差別化要素であるブラウザ自動化が、音声+UIアプリのシステムレベルテストに直接マッチする。AntigravityをDevに配置するとこの能力を活かしきれない。
- **S4 Browser Claude → Customer（シナリオ）**: 多様なユーザー物語の作成、テキストレイヤーでの動作レビューに適している。マイクは物理的に使用できないため…
- **S2 Kosuke → Customer（実音声UAT）**: チーム内のどのAIも実行できない唯一のUATタスク。あえて人間に割り当てることで「このギャップを隠さず文書化する」姿勢を明示する。
- **S8 Claude Cowork → PMO**: Coworkは非エンジニア向けのファイル・タスクワークフロー向けに設計されている。成果物管理を委譲することで、TPMが意思決定に集中できる体制を維持する。

## 職務分離 (Separation of Duties)

- **PO ≠ Customer**: Kosuke（PO）が優先順位を決定、Customerロール（S2 / S4の分割）がユーザーの声を提供する。POがCustomer要望を却下する可能性あり — トレードオフは変更ログに記録する。
- **Dev ≠ QA**: Codex (S6) がコードを書き、Antigravity (S7) が独立してテスト実施（ブラウザ自動化によるシステムテストを含む）。互いの成果物を承認しない。
- **Architect ≠ Dev**: Claude Code (S5) がパターンを定義・レビューし、Codexが実装する。
- **PMO ≠ TPM**: Cowork (S8) が成果物を維持し、TPM (S1) が意思決定とベースライン変更の承認を担う。

## 既知の構造的限界（隠さず文書化）

- Browser ClaudeとBrowser ChatGPTはチャットのみ — ローカルリポジトリを直接参照できない。必要に応じてセッションに関連抜粋を貼付する。
- チームのどのAIもマイク・スピーカーを使えないため、最終の音声UATは人間のみ (S2)。
- 単独の人間オペレーター（バスファクター = 1）— リポジトリに全状態を保持することで軽減（リスクR5参照）。
