# 品質マネジメント / テスト計画書 (Quality Management / Test Plan) — Kotoba

**オーナー:** QA（Antigravity IDE）
**関連文書:** ユーザーストーリー、TDD §4–5、欠陥ログ

## 1. 品質目標

| ID | 目標 | 指標 | 目標値 |
|----|---|---|---|
| Q1 | スキルディスパッチの正確性 | 60ケーステストでの正答率 | ≥ 90% |
| Q2 | スキル別の正しさ | スキルごと20ケースでの合格率 | ≥ 90% |
| Q3 | 音声/UIの整合性 | 10ケース監査での矛盾発生率 | 0% |
| Q4 | レイテンシ | 音声入力 → 音声出力開始（LLMのみのパス） | p50で ≤ 5秒 |
| Q5 | コスト上限 | 日次使用額 | ≤ $2/日（平均） |
| Q6 | コードヘルス | オーケストレータ + スキルのユニットテストカバレッジ | ≥ 70% |

## 2. テストレベル

| レベル | オーナー | スコープ |
|---|---|---|
| ユニット | Dev (Codex) | 純粋関数、スキルロジック、契約バリデータ |
| 結合 | QA (Antigravity) | ベンダーをモック化したAPIレベルのエンドツーエンド |
| 契約 | QA (Antigravity) | `Response`契約: サマリが詳細から導出可能であること |
| システム | QA (Antigravity) | 安価なベンダーに対する音声フルループ + ブラウザ自動化によるUI検証 |
| UAT | Customer（Browser Claude） | ユーザーストーリー単位のシナリオベース |

## 3. テストデータ

- 60ケース・ディスパッチャセット: スキルAが明確 20 / スキルBが明確 20 / スキルなし（フォールバック期待）20
- 20ケース × 3スキル = 60機能ケース
- 音声/UI整合性監査セット 10ケース（エッジケースを厳選）
- 英語 + 日本語の分布を各セットで50/50

## 4. スプリント単位の入口・出口基準

**入口:** スプリント計画がマージ済み、当該スプリントの成果物に対するテストセットの下書きが存在する。
**出口:** スプリント内の全ストーリーに対しパスするテストがある、欠陥が起票済み、ステータスレポートが作成済み。

## 5. 欠陥管理

欠陥は `docs/10-quality/defect-log.md` に記録する。重大度: S1（ブロッカー）/ S2（重大）/ S3（軽微）/ S4（外観）。S1/S2はスプリントレビュー前にクローズすること。

## 6. UAT手順

1. Customer（Browser Claude）がMVPの各ユーザーストーリーを実行する。
2. 各ストーリーに Pass / Fail / Pass-with-notes を付与する。
3. Fail → 欠陥ログ起票 + スコープに関わる場合は変更要求。
4. UATレポートを `docs/11-closing/` に格納し、承認されるまで終結フェーズをブロックする。

## 7. 独立性 (Independence)

QA（Antigravity）はプロダクションコードを書かない。Dev（Codex）はテストの承認権限を持たない。この分離はプログラムの不変条件である。
