# ユーザーストーリー / 要件 — Kotoba

フォーマット: `<ユーザー>として、<能力>を望む、そうすることで<利益>`。INVEST基準。受け入れ基準はGiven/When/Then形式。

ユーザーの声の源: Customerロール = Browser Claude。優先順位付け: TPM/PO = Kosuke、MoSCoW法で実施。

## MVP (Must)

### US-001 — プッシュトゥトーク (PTT) でのクエリ
**ユーザーとして**、キーを押して話しかけたい、**そうすることで**タイプせずに回答を得られる。
- 優先度: M
- Given アプリ起動中、When PTTキーを押しながら話す、Then キー離後1秒以内に音声が転記・送信される。
- 受け入れ基準: 英語・日本語の両方で認識可能。

### US-002 — 音声返答 + 画面上の詳細表示
**ユーザーとして**、短い音声返答に加えて画面に詳細情報を出してほしい、**そうすることで**聞き直さずに深掘りできる。
- 優先度: M
- Given クエリ完了、When アシスタントが返答、Then TTSが30秒以内のサマリを再生し、**かつ**詳細パネルにサマリと整合する構造化データが表示される。
- 受け入れ基準: 10ケースのテストにおいて、音声内容と画面表示内容に矛盾ゼロ。

### US-003 — スキルの呼び出し
**ユーザーとして**、リクエストに応じて正しいスキルを呼び出してほしい、**そうすることで**ハルシネーションではなく実データを得られる。
- 優先度: M
- Given 既知スキルにマッチするクエリ、When 処理、Then 該当スキルが呼び出されその結果が返答生成に利用される。
- 受け入れ基準: 60ケースのディスパッチャテストにおいて ≥ 90% のスキル選択正答率。

### US-004 — 優雅なフォールバック
**ユーザーとして**、マッチするスキルが無くても有用な回答がほしい、**そうすることで**アシスタントが役に立たないと感じない。
- 優先度: M
- Given どのスキルにもマッチしないクエリ、Then LLMが直接回答し、詳細パネルは「一般回答モード」を表示する。

### US-005 — クラウドでの可用性
**ユーザーとして**、別のマシンからもアシスタントに到達したい、**そうすることで**スクリプトではなくサービスとして使える。
- 優先度: M
- Given バックエンドがデプロイ済み、When トークンを持って別マシンのクライアントを開く、Then フルクエリループを完了できる。

## Should

### US-006 — コストの可視性
**運用者として**、日次のAPIコストを記録したい、**そうすることで**月$60の上限を維持できる。
- 優先度: S
- Given 1日使用、Then コストログにスキル別・合計のUSD使用額が表示される。

### US-007 — エラーの透明性
**ユーザーとして**、スキル失敗時にUIでそれが分かってほしい、**そうすることで**アシスタントが嘘をついていると誤解しない。
- 優先度: S
